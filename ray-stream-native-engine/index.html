OmniStream
Ray · ZMQ · Relay 架构

Research prototype，目标是先把 Qwen3-Omni 类多 stage 模型 serve 跑起来。 Ray 只负责启动和生命周期；ZMQ control plane 负责小消息；Relay data plane 负责真实 payload； Worker 负责模型计算；FastAPI / uvicorn 作为 asyncio task 嵌在 GatewayActor 内部， 处理 /v1/chat/completions 与 /v1/realtime。 第一阶段不兼容 vLLM engine 接口，不引入 control / credit / backpressure 协议。

Section 1

三个 plane · 各管一摊

系统拆成三个 plane：control plane（Ray RPC） 启动 actor 和分发 graph； event plane（ZMQ TCP） 在 stage 之间发 ControlPlaneDataEvent 通知； data plane（Relay） 搬 payload 本体。三个 plane 各走各的路。

LIFECYCLE

Ray RPC

低频 · graph / session / actor lifecycle

  • StreamActor: start · shutdown · get_stage（3 个）
  • StageActor: start_stage · stop_stage
  • GatewayActor（stage 的一种，kind="gateway"）: start_stage · open_session · send · recv · close_session · stop_stage · 同时内嵌一个 uvicorn task 暴露 HTTP / WS
  • 不承担 stage-to-stage data
CONTROL PLANE

StageControlPlane

封装 ZMQ TCP PUSH/PULL · 高频 stage 通知

  • 每个 stage（含 gateway）各持一份
  • 底层 PullSocket（TCP bind）/ PushSocket（TCP connect）
  • 每个 stage 的 endpoint 在 StageControlSpec 里 declare
  • wire 上只放 ControlPlaneDataEvent（pickle）
  • 不携带 payload body
DATA PLANE

Relay · LocalRelay

SGLang-style · payload 存取与生命周期

  • put_async / get_async / cleanup / close
  • RelayMetadata 是 ABC，每个 relay 自己 declare subtype
  • v1 实现：LocalRelay（inline · shm）
  • v2：nixl · mooncake · cuda_ipc

Section 2

调用栈

从 HTTP / WS client 到 GPU kernel，OmniStream 由 StreamActor（graph 生命周期）+ N 个 stage actor（其中 kind="gateway" 的那个负责 API protocol + session demux） 组成。每个 actor 都是 Impl + ray.remote(...) 的两层：StageActorImpl 是普通类， StageActor = ray.remote(...)(StageActorImpl)，GatewayActorImpl(StageActorImpl) 通过类继承拿到 stop_stage 等公共 RPC。 uvicorn 作为一个 asyncio task 嵌在 GatewayRuntime 内部，与 _recv_loop 共享同一个事件循环。

Section 3

Remote / ZMQ / Local 边界

看清三类方法：标了 @rpc_method 的是 Ray RPC surface； ZMQ surface 只搬 ControlPlaneDataEvent；其它都是同一进程内的本地方法。

Ray RPC

@rpc_method

跨进程 / 跨 actor · 低频

  • StreamActor: start · shutdown · get_stage（仅 3 个）
  • StageActor: start_stage · stop_stage
  • GatewayActor: start_stage · stop_stage · open_session · send · recv · close_session
  • GatewayActor 的 session RPC 也是 uvicorn handler 直接调的本地 runtime 方法（同进程，无 RPC hop）；Ray RPC 入口主要给单元测试 / 旧 client 用
  • StageActor / GatewayActor 不提供 data / control / credit receive RPC
ZMQ surface

ControlPlaneDataEvent · 唯一通过

stage-to-stage event plane

  • wire 上只有 ControlPlaneDataEvent（pickle, TCP）
  • 不带 route 字段：目的 stage 由 endpoint 决定
  • 不带 src 字段：源 stage 是当前发送方
  • chunk_metadata 只塞 modality / seq_no / edge_id
  • relay_metadata 是 relay 后端返回的 opaque handle
Local-only

普通 Python 方法

同 actor 进程内 · 不暴露到 Ray

  • StreamRuntime.*（StreamActorImpl 的真实实现）
  • StageRuntime._recv_loop · _send_loop · _handle_event · _send_output
  • GatewayRuntime._recv_loop · _serve_api · _handle_event · send · recv · open_session · close_session
  • build_app(runtime) · ChatServing.create / _stream / _nonstream · RealtimeServing.handle（API 层，由 uvicorn task 驱动）
  • Worker.run
  • Relay.put_async · get_async · cleanup · close
  • RelayOperation.wait_for_completion
  • Worker.load_model · run · unload_model

两条 Loop + Worker.run 的分工

_recv_loopin StageRuntime
  • control.recv()（ZMQ PULL）
  • op = await relay.get_async(event.relay_metadata, request_id=event.session_id)
  • await op.wait_for_completion()
  • relay.cleanup(event.session_id)
  • input_queue.put(WorkerInput)
Worker.runin Worker
  • input_queue.get()
  • engine loop（batching / scheduling / 流式输出，由 Worker 自治）
  • output_queue.put(WorkerOutput)
  • v1 FakeWorker 每次一个 item 直传；下一步 VllmWorker 包 AsyncLLMEngine
_send_loopin StageRuntime
  • output_queue.get()
  • routes_by_port[output.port] 选路
  • op = await relay.put_async(output.payload, request_id=output.session_id)
  • control.send_to_stage(dst, stage_specs[dst].control.connect_endpoint, event)
  • await op.wait_for_completion()
  • 无 route 时丢弃（sink → gateway 现在是正常 Route，不再需要 fallback）
Cheat sheet。 被 @rpc_method 标 → Ray RPC； 出现 ZMQ socket → event plane； 其它一律本进程 asyncio task / async method。
Section 4

组件细节

逐层 ×（actor / runtime）的真实接口。所有签名都对应 ~/code/omnistream 当前源码。

4.1 · StreamActor — graph lifecycle 边界

thin Ray actor，所有 @rpc_method 一行 delegate 到内部 StreamRuntime。 RPC surface 收窄到 3 个：start / shutdown / get_stage。 所有 session / data 相关的 RPC（open_session / send / recv / close_session） 都搬到了 gateway stage 上。

class StreamActorImpl: def __init__(self): self.runtime = StreamRuntime() @rpc_method async def start(self, graph: GraphSpec) -> RpcAck: return await self.runtime.start_graph(graph) @rpc_method async def shutdown(self) -> None: await self.runtime.shutdown() @rpc_method async def get_stage(self, stage_id: str) -> Any: return self.runtime.stage_actors[stage_id] StreamActor = ray.remote(max_concurrency=128)(StreamActorImpl)

4.2 · StreamRuntime — graph lifecycle only

不是 Ray actor；普通 class。瘦身版：只持 graph 和 stage_actors 字典。 没有 RuntimeControlPlane、没有 LocalRelay、没有 session table、没有 ingress relay.put。所有这些都搬到 gateway stage 里去了。

class StreamRuntime: """Owns graph topology and stage actor lifecycle. Nothing else.""" def __init__(self): self.graph: GraphSpec | None = None self.stage_actors: dict[str, Any] = {} async def start_graph(self, graph: GraphSpec) -> RpcAck: ... async def shutdown(self) -> None: ...

start_graph 做的事

  • 从 graph.routes 编译每个 src stage 的 routes tuple。
  • 对每个 stage 根据 spec.kind 选 actor 类： actor_cls = GatewayActor if spec.kind == "gateway" else StageActor， 然后 actor_cls.options(num_cpus=0).remote()。
  • 用 asyncio.gather 并行启动所有 stage。
    普通 stage：actor.start_stage.remote(spec, routes, stage_specs)；
    gateway stage：额外把 ingress=graph.ingress 关键字一起传过去。
    stage_specs（全量 stage 字典）一并传给每个 stage，方便 stage 自己查下游 control endpoint。

shutdown 做的事

  • 对所有 stage actor 调 stop_stage.remote()。
  • 清空 stage_actors。

4.3 · StageActor — Ray RPC lifecycle wrapper

每个物理 stage 一个 Ray actor。RPC 表面只剩 start_stage / stop_stage 两个。 实现拆成 StageActorImpl（普通类）+ StageActor = ray.remote(...)(StageActorImpl) —— 这样 GatewayActorImpl 可以直接 class GatewayActorImpl(StageActorImpl) 继承公共 stop_stage、self.runtime 等结构。

class StageActorImpl: """Plain class form of the stage actor. Inherit this to extend, then wrap with ray.remote(...). """ def __init__(self): self.runtime: StageRuntime | None = None @rpc_method async def start_stage( self, spec: StageSpec, routes: tuple[Route, ...], stage_specs: dict[str, StageSpec], ) -> RpcAck: ... @rpc_method async def stop_stage(self) -> None: ... StageActor = ray.remote(max_concurrency=128)(StageActorImpl)
注意。 ① 没有 set_routes RPC 了 — routes 在 start_stage 时一次传完；
② 没有 stage-output 拉取 RPC 了 — sink 通过普通 Route 把结果回送到 gateway，gateway 自己 demux；
③ 没有任何 recv_data_event / recv_control_event Ray RPC handler — stage 间通信只走 ZMQ。

4.4 · StageRuntime — 本 stage 真理源

持有 StageControlPlane（PULL bind + 下游 PUSH cache）、LocalRelay、 两个 queue、routes_by_port 路由表、对全量 stage_specs 的视图（为了查下游 endpoint）和两条 asyncio loop + Worker.run task。

能力方法 / 字段说明
recv _recv_loop() · _handle_event() control.recv() 拉 event；调 relay.get_async + wait_for_completion + relay.cleanup(session_id)；写 input_queue。
relay relay: LocalRelay 本 stage 自己的 relay 实例；按 request_id 跟踪 ref 生命周期。
queues input_queue · output_queue unbounded asyncio.Queue。
routing routes_by_port · stage_specs output port → routes 表；下游 endpoint 通过 stage_specs[dst].control.connect_endpoint 查。
send _send_loop() · _send_output() 从 output_queue 取，relay.put_async 后用 control.send_to_stage 发给下游。无 route 时丢弃（sink 通过普通 Route 回送到 gateway）。

StageRuntime 最小代码形态

class StageRuntime: async def _recv_loop(self) -> None: while not self._stopping.is_set(): event = await self.control.recv() await self._handle_event(event) async def _handle_event(self, event: ControlPlaneDataEvent) -> None: op = await self.relay.get_async(event.relay_metadata, request_id=event.session_id) await op.wait_for_completion() payload = op.payload self.relay.cleanup(event.session_id) await self.input_queue.put( WorkerInput( session_id=event.session_id, payload=payload, chunk_metadata=dict(event.chunk_metadata), ) ) async def _send_loop(self) -> None: while not self._stopping.is_set(): output = await self.output_queue.get() await self._send_output(output) async def _send_output(self, output: WorkerOutput) -> None: routes = self.routes_by_port.get(output.port, []) for route in routes: op = await self.relay.put_async(output.payload, request_id=output.session_id) event = ControlPlaneDataEvent( session_id=output.session_id, relay_metadata=op.metadata, chunk_metadata={**output.chunk_metadata, "edge_id": route.edge_id}, ) endpoint = self.stage_specs[route.dst_stage].control.connect_endpoint await self.control.send_to_stage(route.dst_stage, endpoint, event) await op.wait_for_completion()

4.5 · Worker — engine loop owner

本 stage 的计算抽象。Worker 自己拥有 engine loop：StageRuntime 在 start() 阶段把 Worker.run 作为一个 asyncio task 挂到 input/output queue 之间，stop() 时由 task.cancel() 终止。 worker 内部决定 batching / scheduling / 流式输出等策略，runtime 完全不关心。

class Worker(ABC): async def load_model(self, spec: StageSpec) -> None: ... async def run( self, input_queue: asyncio.Queue[WorkerInput], output_queue: asyncio.Queue[WorkerOutput], ) -> None: ... async def unload_model(self) -> None: ... class FakeWorker(Worker): # validates runtime wiring before real model_runner integration async def run(self, input_queue, output_queue): while True: item = await input_queue.get() await output_queue.put(self._transform(item))
为什么 executor owns the loop？ vLLM 的 EngineCoreProc、SGLang 的 scheduler 都是「executor 自己拥有 loop」。 之前如果把 loop 留在 StageRuntime 里、用 run_batch(batch) -> list 接口反向调 worker， 接 vLLM 就要不停地和它内部的 continuous batching 打架。queue-driven 的 Worker.run 让 vLLM、SGLang、自写 batcher 都能直接落到这一层。

4.6 · Gateway — API 协议 + session demux 的特殊 stage

Gateway 是 graph 里一个普通的 stage（spec.kind == "gateway"），只是它的"worker"不是模型， 而是 session 多路解复用 + API 协议翻译。结构上和 StageRuntime 同构： 自己持 StageControlPlane（PULL bind + PUSH cache）、自己的 LocalRelay、自己的 recv loop。 区别在于 _handle_event 把收到的 payload 按 session_id 投递到 session_outputs: dict[sid, asyncio.Queue]，而不是 worker 的 input_queue。 sink stage 通过普通 Route（sink → gateway）把结果送过来——没有"egress"特殊通路。

v7 起，gateway 还多了一个 _serve_api 任务：当 spec.config["api_port"] 不为 None 时， GatewayRuntime.start() 在 _recv_loop 旁边再 create_task 起一个 uvicorn server， build_app(self) 直接把 runtime 自己传给 FastAPI。uvicorn config 用 loop="asyncio" + lifespan="off"， 复用 actor 的事件循环。详见 Section 5 · API 层。

两种典型客户端调用。
① HTTP / WS（v7 起推荐）： stream_actor.start.remote(graph)（其中 gateway stage 的 spec.config["api_port"] 已设） → uvicorn 起来后 POST http://host:port/v1/chat/completions 或 WS /v1/realtime， ChatServing / RealtimeServing 在 actor 内部自动 open_session / send / recv / close_session。

不论走哪条路径，gateway 内部都按 session_id 维护一组 asyncio.Queue，多 session 并发不会串台。

4.7 · 一个 StageActor 内部全貌

Section 5

API 层 · uvicorn 嵌在 GatewayActor 内

v7 起 graph 直接对外暴露 HTTP / WS：FastAPI app 由 build_app(runtime) 构造， uvicorn.Server 作为 asyncio task 跑在 GatewayActor 内部、与 _recv_loop 共用同一个事件循环。 Handler 拿着 GatewayRuntime 的引用，直接调 runtime.send / recv / open_session / close_session —— 不再走 actor.method.remote，没有 Ray RPC hop，没有 IPC。

5.1 · uvicorn 挂在哪 · 为什么 colocate

GatewayRuntime.__init__ 多了 api_port / api_host 两个参数（来自 spec.config）；start() 在 _recv_loop 旁边 create_task(_serve_api)。 _serve_api 用 uvicorn.Config(app, loop="asyncio", lifespan="off") —— 这两个选项强迫 uvicorn 复用 actor 已经在跑的事件循环，而不是自己起一个。 这样 handler 里的 await runtime.send(...) 和 _recv_loop 里的 await control.recv() 是同一个 loop 里的两个 task，session_outputs[sid] 是同一个 asyncio.Queue，recv 端 await q.get() 拿到的就是 _handle_event 刚 put 进去的对象。

为什么不另起进程？ 也可以让 API server 单独是一个 Ray actor 或裸进程，然后用 Ray RPC 调 gateway —— 但那样每条 HTTP / WS 请求都要走一次跨进程 RPC， 多一次 pickle / hop。把 uvicorn 塞进 gateway 是 vLLM / sglang / vllm-omni 都采用的做法（同一进程内 protocol layer + engine），off-loop 也方便 handler 直接共享 runtime 状态（sessions 集合、session_outputs 字典）。

ChatServing skeleton

class ChatServing: def __init__(self, runtime: GatewayRuntime): self.runtime = runtime async def create(self, req: ChatCompletionRequest): if req.stream: return StreamingResponse(self._stream(req), media_type="text/event-stream") return await self._nonstream(req) async def _nonstream(self, req) -> ChatCompletion: sid = f"chat-{uuid.uuid4().hex}" await self.runtime.open_session(sid) try: await self.runtime.send(sid, "text", {"text": self._last_user_text(req), "path": []}) output = await self.runtime.recv(sid, timeout=10.0) finally: await self.runtime.close_session(sid) if output is None: raise HTTPException(504, "graph produced no output within 10s") return ChatCompletion(...) async def _stream(self, req) -> AsyncIterator[str]: sid = f"chat-{uuid.uuid4().hex}" await self.runtime.open_session(sid) try: await self.runtime.send(sid, "text", {"text": ..., "path": []}) yield frame(role="assistant") # opener output = await self.runtime.recv(sid, timeout=10.0) yield frame(content=format(output.payload)) # body yield frame(finish_reason="stop") # closer yield "data: [DONE]\n\n" finally: await self.runtime.close_session(sid)

RealtimeServing skeleton

class RealtimeServing: async def handle(self, ws: WebSocket) -> None: await ws.accept() sid = f"rt-{uuid.uuid4().hex}" await self.runtime.open_session(sid) await ws.send_json({"type": "session.created", "session": {"id": sid, ...}}) async def reader(): while True: event = await ws.receive_json() await self._handle_client_event(event, sid) # → runtime.send async def drainer(): while True: output = await self.runtime.recv(sid, timeout=3600.0) if output is None: return await ws.send_json(self._server_event(output)) r, d = asyncio.create_task(reader()), asyncio.create_task(drainer()) try: done, pending = await asyncio.wait({r, d}, return_when=asyncio.FIRST_COMPLETED) for t in pending: t.cancel() except WebSocketDisconnect: pass finally: await self.runtime.close_session(sid)
为什么要写成 class？ ① 一旦 endpoint family 出现 per-handler 状态（token counter / rate-limit / per-session 模板渲染器）， class 是天然的安插点，不会让 server.py 长成 god module； ② 单测时不用启动 uvicorn，直接 ChatServing(mock_runtime) 就能调； ③ 想加 streaming 变种就 class StreamingChatServing(ChatServing)，路由表不动； ④ 这就是 vLLM / sglang / sgl-omni 都在用的 layout，对照阅读 / 迁移 prompt 模板成本最低。
Session id 的 demux 语义 · 关键。 不管走 HTTP 还是 WS，每个请求一开始都 sid = "chat-{uuid}" 或 "rt-{uuid}"。 这个 sid 由 runtime.send 盖在 ControlPlaneDataEvent.session_id 上， 被每个中间 stage 原样从 WorkerInput.session_id 复制到 WorkerOutput.session_id， 最后被 gateway 的 _handle_event 拿来在 session_outputs dict 里查队列。 中间的 ingress / thinker / sink 完全不读、不分支、不依赖 session_id —— 它们只是 stateless 计算 + 透传。 只有两端（runtime.send 这一端写、gateway._handle_event 这一端读）真正用到这个字段。 这是 vLLM RequestTracker + request_id、sglang rid_to_state 的同一个套路。
Section 6

事件结构

v1 只有一种 stage-to-stage 事件：ControlPlaneDataEvent。 没有 ControlEvent、没有 CreditEvent、没有 BARGE_IN 多播 — 这些等 minimal serve 跑通后再考虑。

6.1 · ControlPlaneDataEvent

@dataclass(frozen=True) class ControlPlaneDataEvent: session_id: str relay_metadata: RelayMetadata # ABC; LocalRelay 下是 LocalRelayMetadata chunk_metadata: dict[str, Any] = field(default_factory=dict)

没有 route 字段：目的 stage 已经由 ZMQ endpoint 决定，源 stage 是当前发送方。 relay_metadata 是 Relay 后端返回的 handle，下游用它再 relay.get_async 拿真实 payload。 chunk_metadata 只塞 modality / seq_no / edge_id， 不塞 scheduling / control / SLO / deadline 协议。

  • modality: str（决定 entry stage）
  • payload: Any
  • meta: dict | None

  • session_id
  • payload（已 relay.get_async 出来）
  • chunk_metadata

WorkerOutput

Control Plane · TCP ZMQ

v4 起把 IPC 换成 TCP，让 control plane 可以跨节点。 每个 stage 在 StageControlSpec 里 declare 自己的 listen_host:port 和 connect_host:port。 runtime 和上游 stage 用 connect_endpoint 连过来。

7.1 · StageControlSpec

@dataclass(frozen=True) class StageControlSpec: listen_host: str listen_port: int connect_host: str connect_port: int @property def listen_endpoint(self) -> str: return f"tcp://{self.listen_host}:{self.listen_port}" @property def connect_endpoint(self) -> str: return f"tcp://{self.connect_host}:{self.connect_port}"

典型多机配置：

StageActor on node N listen: tcp://0.0.0.0:19000 # PullSocket bind connect: tcp://10.1.2.3:19000 # peers connect here

7.2 · PullSocket / PushSocket（低层）

PullSocket

stage-local · PULL · TCP bind

  • 构造时传 StageControlSpec，从中取 listen_endpoint 做 bind
  • connect_endpoint 暴露给外部，告诉上游往哪连
  • start() · recv() → pickle.loads → ControlPlaneDataEvent
  • close()
  • 共享 zmq.asyncio.Context.instance()

PushSocket

PUSH · TCP connect 单一 endpoint

  • 构造时传 endpoint: str
  • connect()：socket.connect(endpoint)
  • send(event)：pickle.dumps(event, HIGHEST_PROTOCOL)
  • close()

7.3 · StageControlPlane（上层）

StageControlPlane

每个 stage 各一份（包括 gateway）· PULL bind + PUSH cache

  • __init__(stage_id, *, control: StageControlSpec)
  • start() → recv() 拉 ControlPlaneDataEvent
  • send_to_stage(stage_id, endpoint, event)：缺 PushSocket 就新建 connect，缓存在 _stage_sockets[stage_id]
  • connect_endpoint 属性暴露本 stage 的连接地址
  • close() 关 PULL + 全部下游 PUSH socket

没有 RuntimeControlPlane

v6 删掉了

  • 之前 StreamRuntime 还有一个只 PUSH 的 control plane 用来推 entry event
  • 现在 entry event 由 gateway stage 用自己的 StageControlPlane.send_to_stage 推
  • StreamRuntime 完全离开数据路径，不再持有任何 socket

7.4 · wire 上有什么

消息序列化方向说明
ControlPlaneDataEventpickle (HIGHEST_PROTOCOL)上游 TCP PUSH → 下游 TCP PULL唯一 ZMQ payload；带 RelayMetadata
———v1 没有别的 wire message：ControlType / CreditEvent / heartbeat 都不在面上
安全。 Control message 用 pickle 序列化，只能跑在受信任内网； 暴露到不可信网络前需要加认证或换安全序列化格式。
跨节点能力。 Control plane 现在 TCP，理论上能跨机器。 但 payload 能否跨机器还要看 relay backend — 当前 LocalRelay 的 shm 模式仍然只适合同机； 跨节点 payload 需要后面接 nixl / mooncake / nccl relay。

SGLang-style：put / get 返回 RelayOperation，按 request_id 管理 ref 生命周期。 v4 把代码拆成 relay/base.py（接口）和 relay/local.py（v1 实现）。 RelayMetadata 是 ABC，每个 relay 自己 declare metadata subtype。

8.1 · 模块拆分

8.3 · LocalRelayMetadata

8.4 · LocalRelay v1 backend 规则

inline

小 CPU 对象 · 默认

  • payload 不是 bytes/bytearray/memoryview，或大小 ≤ 64 KiB
  • handle = payload 本身
  • get_async() 返回 RelayOperation(metadata=ref, payload=ref.handle)
  • cleanup() 对 inline 是 no-op

shm

大 CPU bytes · multiprocessing.shared_memory

  • payload 是 bytes/bytearray/memoryview 且大小 > 64 KiB
  • handle = {"name": shm.name, "size": nbytes}
  • get_async() 重 attach 读出 bytes 并 close
  • cleanup() 调 shm.unlink() + close()，幂等

8.5 · ref 生命周期与 request_id

LocalRelay._refs_by_key: dict[str, list[LocalRelayMetadata]]： put / get 都把 ref 同时挂到 payload_id 和 request_id 两个 key 上。 cleanup(request_id) 把 key 下所有 ref 释放（shm unlink），并从 payload_id 索引里摘掉。

当前 prototype 把 request_id 用作 session_id。 _handle_event 一拿到 payload 就 cleanup(event.session_id)； _send_output 用 output.session_id 作 put 的 request_id。 这种 1-event = 1-cleanup 适合 v1 单消费 / 无 fan-out； fan-out 或多消费场景需要重新设计（例如改用 turn_id / chunk_id 作 request_id，或加 refcount）。

9.1 · 文字版分解

1

API client → StreamActor（启动 graph）

客户端先 await stream_actor.start.remote(graph) 启动 graph，再 gateway = await stream_actor.get_stage.remote("gateway") 拿到 gateway 的 actor handle。

ack = await stream_actor.start.remote(graph) · gateway = await stream_actor.get_stage.remote("gateway")
2

API client → GatewayActor.open_session

await gateway.open_session.remote(sid)：GatewayRuntime 在 session_outputs 里创建一个空 asyncio.Queue。

3

API client → GatewayActor.send

await gateway.send.remote(sid, modality, payload, meta)。Gateway 按 modality 查 entry stage。

op = await self.relay.put_async(payload, request_id=sid) · ControlPlaneDataEvent(session_id=sid, relay_metadata=op.metadata, chunk_metadata={**meta, "modality": modality})
4

GatewayRuntime: TCP PUSH 到 ingress

用 gateway 自己的 StageControlPlane PUSH 到 entry_spec.control.connect_endpoint；然后 await op.wait_for_completion()。StreamRuntime 不在路径上。

await self.control.send_to_stage(entry_stage, entry_spec.control.connect_endpoint, event)
5

ingress StageRuntime: _recv_loop / _handle_event

control.recv() 拉 event；relay.get_async + wait_for_completion + relay.cleanup(session_id)；input_queue.put(WorkerInput)。

6

ingress Worker: Worker.run

await input_queue.get() → 模型计算 → output_queue.put(WorkerOutput)。

7

ingress StageRuntime: _send_loop / _send_output

按 routes_by_port[output.port] 选下游；relay.put_async → endpoint = stage_specs[dst_stage].control.connect_endpoint → control.send_to_stage(dst, endpoint, event) → op.wait_for_completion。

8

thinker / sink 重复 5–7 步

thinker 下游 = sink；sink 下游 = gateway（普通 Route）。

9

GatewayRuntime: _recv_loop / _handle_event

Gateway 把 sink 推过来的 event 拆出 payload 包成 WorkerOutput，按 session_id 投递到 session_outputs[sid] 队列。

10

API client ← GatewayActor.recv

output = await gateway.recv.remote(sid, timeout)。客户端用完一个 session 后 gateway.close_session.remote(sid)；shutdown 时调 stream_actor.shutdown.remote()。

Section 11

现在不做的事

v1 是 minimal serve prototype。下面这些早期设计稿里有的概念全部砍掉， 等 Qwen3-Omni text/audio smoke 跑起来再决定是否补回来。

不在 v1 内

  • ControlType / ControlEvent · BARGE_IN / CANCEL_EPOCH / FLUSH
  • CreditEvent · receiver-driven credit
  • edge bounded credit / inflight bytes（max_inflight_bytes 只 declared）
  • epoch / turn_id / barge-in
  • EDF / deadline / SLO class
  • batch policy（每次 run([item]) 单条）
  • fan-in / join 等齐多上游
  • per-edge payload projection
  • HTTP / WS / gRPC 协议适配（gateway 已经有结构，但实际协议层还没写）
  • cuda_ipc / nixl / mooncake relay
  • placement-aware config（手填 listen_host / connect_host）

v1 已有

  • Ray actor 启动 / 生命周期：StreamActor（3 RPC）+ StageActor + GatewayActor
  • Impl + ray.remote 拆法：StageActorImpl / StreamActorImpl / GatewayActorImpl(StageActorImpl)
  • StreamRuntime 瘦身（只 graph + stage_actors）/ StageRuntime / GatewayRuntime 拆层
  • Gateway 作为一个普通 stage：API 协议翻译 + session_outputs 多路解复用
  • TCP control endpoint · StageControlSpec
  • StageControlPlane（gateway 和普通 stage 都用同一个）
  • PUSH/PULL ControlPlaneDataEvent
  • SGLang-style Relay ABC + LocalRelay（inline / shm）
  • RelayMetadata ABC + LocalRelayMetadata
  • linear graph ingress → thinker → sink → gateway（sink→gateway 是普通 Route）
  • FakeWorker end-to-end smoke test（gateway.send / recv）
  • 两条 loop（_recv / _send）+ Worker.run（普通 stage）；1 条 recv loop（gateway）
  • @rpc_method 标注 + list_rpc_methods introspection
设计目标提醒。 stage-to-stage 数据路径上完全不经过中心 runtime，也不走 Ray object store。 API ingress / egress 由 gateway stage 承担 —— 它本身就是 graph 里的一个 stage， 用同一套 StageControlPlane + Route 接入图里，StreamRuntime 只负责启动它。
Section 12

下一步实现顺序

从 Qwen3-Omni 真模型接入开始，逐步补 fan-in、terminal channel、跨节点 relay， 最后跑 Qwen3-Omni / TTS / interactive workload 的 latency / RTF / goodput 实验。

1

Qwen3-Omni executors

preprocess · audio / image encoder · thinker · talker · codec executor，逐个接到 Worker 接口上。 对照 third_party/vllm-omni 和 third_party/sglang-omni 的 model runner 边界。

2

Fan-in / join stage

支持 encoder 输出汇聚到 thinker 等多输入 stage，等齐上游再 run。

3

真实 API 协议接到 gateway

当前 GatewayActor.send / recv 只是 Ray RPC。下一步在它前面接 HTTP / WS / gRPC server， 把请求 demux 成 send / 把 recv 出来的 WorkerOutput 转成 streaming chunk。

4

跨节点 payload relay

先 nixl / mooncake（cross-node），同机 GPU 再补 cuda_ipc。 RelayMetadata ABC 已经预留好扩展点。

5

Qwen3-Omni / TTS / interactive 实验

在真实 workload 上做 latency / RTF / goodput 实验，根据 traces 再决定是否回头补 control / credit / BARGE_IN。

在同一个 EngineCore
调度 AR、DiT、artifact

UAD (Unified AR + DiT) 在 vLLM v1 EngineCore 进程边界内构建 unified engine, 让同一个 DP group 可以同时调度 AR token、DiT denoise step 和 artifact decode。 第一目标模型是 HunyuanImage3。 当前代码是 scaffold, 不是完整 serving path——这份文档说清楚 v1 EngineCore protocol 保持哪些不变, UAD 在哪些边界上做覆盖, 以及 state / commit / attention 的语义如何被严格定义。

01
§ 01 · 接入

从 CLI 到 UADEngineCore

入口由 vLLM-Omni serve CLI 触发, 通过环境变量与 stage-startup 决定 EngineCore 类型, 最终让 stage process 用 UAD 版本替代标准 v1 EngineCore。

CLI → process → engine core
关键决定: 不在 worker / model_runner 层做切换, 而是直接替换 EngineCore 类。 这样 step 调度循环、scheduler/executor 组合、output 处理全部在一个对象内被覆盖, 上层 multiprocessing 和 lifecycle 路径完全沿用 v1。 
# 启动链
vllm-omni serve --uad-engine
  → os.environ["VLLM_OMNI_USE_UAD_ENGINE"] = "1"
  → StageEngineCoreProc.run_stage_core()
  → if uad_engine_enabled(): engine_cls = UADEngineCore
  → UADEngineCore(StageEngineCoreProc)  # 替换标准 v1 EngineCore
02
§ 02 · 对象边界

两条平行路径在 一个 进程里

UADEngineCore 同时持有 v1 原生 scheduler / executor (给 EngineCoreProc lifecycle 用) 和 UAD-native scheduler / executor / runner (UAD step path 用)——这是一个 narrow facade, 不是 wrapper。

UADEngineCore 内部对象表
UAD 不持有 base_scheduler / base_executor—— 这是从 a55947aa 到 61934ecd 的一个明确清理: 早期版本曾让 UAD facade 同时持有并 delegate base 实例, 后来简化为 UAD-native scaffold, v1 实例继续由 EngineCoreProc.scheduler / self.model_executor 拥有, 互不混淆。
对象角色所有者
self.scheduler原生 v1 scheduler — 保留给 EngineCoreProc lifecycle 路径v1
self.model_executor原生 v1 executor — 保留给 EngineCoreProc lifecycle 路径v1
self.uad_schedulerUAD step path 的 scheduler scaffoldUAD
self.uad_executorUAD step path 的 executor scaffoldUAD
self.uad_runnerUAD output processor boundaryUAD
03
§ 03 · 类关系

继承用于接口可读性, 不用于复用

UAD 类继承 v1 interface / base class, 只用于接口可读性和 abstract method 检查; UAD 不再持有 base_scheduler 或 base_executor。

UAD class hierarchy
继承 ≠ 复用: UAD facade 只用 v1 父类做接口可读性, 实际行为完全 native。 UADScheduleItem、UADRequestState、UADPhaseOutput 是纯 UAD 概念, 没有 v1 等价物, 因此不挂任何父类。
04
§ 04 · STEP FLOW

一次 step() 的七步

UADEngineCore.step() 替换 v1 的 step orchestration——schedule → execute → grammar → sample → process → update, 按顺序覆盖 AR、DiT、artifact 三类 work, 保留 sample tokens 作为 AR 专属的 optional 步骤。

UADEngineCore.step() orchestration
当前 scaffold 行为: has_requests() 返回 False; schedule() 返回 empty UADSchedulerOutput; execute_model() 返回 no-op future / None; sample_tokens() 返回 empty UADModelRunnerOutput。 整条链已经被 wired up, 但单元 step 实现是后续 commit 的 TODO。
05
§ 05 · SCHEDULER OUTPUT

v1 字段 + uad_items

UADSchedulerOutput 是对 v1 SchedulerOutput 的 additive 扩展: AR 字段语义不变, 新增轻量 uad_items metadata 列表用于 DiT / artifact work。 

@dataclass
class UADSchedulerOutput(SchedulerOutput):
    uad_items: list[UADScheduleItem]
v1 inherited 字段

AR work — 复用原语义

num_scheduled_tokens, num_common_prefix_blocks, scheduled_new_reqs, scheduled_cached_reqs, ...

AR prefill / decode 完全沿用 v1 chunked-prefill / decode 语义。 不引入新的 phase budget。

UAD 新增字段

uad_items — DiT / artifact

每个 UADScheduleItem 描述一个 DiT step 或一次 artifact decode, 仅放轻量 metadata。

✗ 不放大 tensor
✗ 不放 CUDA tensor
✓ 仅 request_id / phase / shape bucket / step index

约束: 同一个 UADSchedulerOutput 可以同时包含 AR work 和 DiT/artifact items—— 这是 unified engine 的关键, executor 应在兼容层做合批。

06
§ 06 · WORK ITEM

每个 DiT step 是一个 UADScheduleItem

UADScheduleItem 是 frozen dataclass, schedule 时一次性写入, 携带 attention 与 commit 行为所需的全部 metadata。 

@dataclass(frozen=True)
class UADScheduleItem:
    request_id: str
    phase: UADPhase
    num_scheduled_tokens: int
    input_kind: Literal["ar_tokens", "dit_latent_step", "artifact_decode"]
    output_kind: Literal["sample_tokens", "denoise_pred", "artifact", "none"]
    needs_kv_slots: bool = False
    num_persistent_tokens: int = 0
    step_index: int | None = None
    total_steps: int | None = None
    shape_bucket: tuple[int, ...] | None = None
字段写入时机含义
num_scheduled_tokensschedule本 tick 的 token-like compute budget
num_persistent_tokensschedule本 item 成功后预计 commit 到 reusable engine context / KV 的 token 数
needs_kv_slotsscheduleexecutor 是否需要物理 KV slot / slot mapping
step_index / total_stepsscheduleDiT step batching metadata
shape_bucketscheduleDiT shape batching metadata
DiT step 的 token 语义
DiT non-final step 通常 num_scheduled_tokens > 0、num_persistent_tokens = 0。 DiT final image-context commit 通常 num_persistent_tokens = image context token count。
07
§ 07 · REQUEST STATE

每个 request 的 UAD ledger

UADRequestState 是 scheduler-owned, 维护 phase 进度、统一 token ledger 和模型私有状态。 AR sampler history 不复制——继续由 v1 字段维护。 

class UADPhase(str, Enum):
    AR_PREFILL = "ar_prefill"
    AR_DECODE = "ar_decode"
    DIT_STEP = "dit_step"
    ARTIFACT_DECODE = "artifact_decode"

@dataclass
class UADRequestState:
    request_id: str
    phase: UADPhase
    engine_tokens: list[UADToken]
    materialized_tokens: list[UADToken]
    dit_step_index: int
    dit_num_steps: int
    dit_query_tokens: int
    runtime_state: dict[str, Any]
字段含义
phase请求当前阶段 (AR_PREFILL / AR_DECODE / DIT_STEP / ARTIFACT_DECODE)
engine_tokens后续模型可见的统一多模态 logical token ledger
materialized_tokens对外 streaming / artifact output ledger
dit_step_index / dit_num_stepsDiT 进度
dit_query_tokens本图像 context 的 token-like 数量
runtime_state模型私有状态: image size、seed、CFG、latent shape
UAD 不维护独立 ar_sampler_token_ids。 AR sampler history 继续由 v1 request / scheduler / model runner 原生字段维护。 — design_uad.md §6

§ 08 · PHASE OUTPUT

Runner 只返回 delta, 不改 state

Runner output 是只读 delta; scheduler 在 step 7 接收并 apply, 这样 ownership 不混。 

@dataclass
class UADPhaseUpdate:
    new_engine_tokens: list[UADToken]
    new_materialized_tokens: list[UADToken]
    num_new_computed_tokens: int
    next_phase: UADPhase | None
    runtime_state_delta: dict[str, Any]

@dataclass
class UADPhaseOutput:
    request_id: str
    phase: UADPhase
    update: UADPhaseUpdate
    raw_output: Any | None = None
字段写入时机含义
new_engine_tokensrunner output新增到 UAD logical engine ledger 的 token
new_materialized_tokensrunner output新增到对外输出 ledger 的 token / artifact
num_new_computed_tokensrunner output实际推进 reusable computed-prefix / KV 语义的 token 数
next_phaserunner outputphase transition 目标
runtime_state_deltarunner output模型私有状态增量

第一版可要求 num_new_computed_tokens == UADScheduleItem.num_persistent_tokens。 后续支持 partial commit / preemption 时二者可以不同。

§ 09 · 状态所有权

谁拥有哪段状态

整个 UAD 设计的核心矛盾在于: 怎么让 v1 的 sampler / KV / num_computed_tokens 路径继续工作, 同时让 UAD 拥有 phase / engine_tokens / materialized 的新概念。下表把 ownership 写清楚。

状态Owner
Request.prompt_token_idsv1 request
v1 generated token ledgerv1 scheduler / request
Request.num_computed_tokensv1 scheduler / KV path
SchedulerOutput.num_scheduled_tokensv1 scheduler
ModelRunnerOutput.sampled_token_idsv1 model runner / sampler
UADRequestState.phaseUAD scheduler / state machine
UADRequestState.engine_tokensUAD state
UADRequestState.materialized_tokensUAD state / output path
UADScheduleItem.num_persistent_tokensUAD scheduler

num_computed_tokens 只在 token 真正 commit 到后续可复用 engine state / KV 后推进。 一次 forward 不必然增加 computed tokens。 — design_uad.md §8

§ 10 · STATE MACHINE

Phase 是下一次可调度 work 的类型

UAD request state machine 是 scheduler-owned。phase 只表示下一次可调度 work 的类型, 而不是每个 Python 后处理函数都要变成 scheduler-visible phase。

UAD phase transition graph
ARTIFACT_DECODE 类似输出 materialization, 不等价于 v1 text sampling。 它是否由 scheduler 感知取决于是否占用 GPU / model budget; 纯 CPU / IO 后处理可以从 scheduler 隐藏。

§ 11 · SCHEDULER 语义

统一在 token-like budget 之下

UAD scheduler 仍按 token-like budget 工作, 不引入独立的 phase budget——这样 AR 和 DiT 能在同一个调度循环里 被一致比较。

Work表示方式调度粒度
AR prefill / decodeinherited SchedulerOutput 字段复用 v1 chunked prefill / decode 语义
DiT stepUADScheduleItem一个 denoise step 是一个调度单元
artifact decodeUADScheduleItem 或 scheduler 外后处理取决于是否占 GPU / model budget
第一版不把一个 DiT denoise step 再切成任意小 chunk —— timestep、latent、CFG 和 image query 是耦合的; 除非具体模型显式支持 partial step。 — design_uad.md §10

同一个 UADSchedulerOutput 可以同时包含 AR work 和 DiT / artifact uad_items。 executor 可以按 attention 约束拆 kernel, 但 FFN / MoE 只要 hidden size、dtype、layer、TP / EP mesh 和 backend 兼容, 就应尽量合批——这是 unified engine 的吞吐来源。

12
§ 12 · KV / PERSIST COMMIT

三个 token 数, 三个不同时机

num_scheduled_tokens 是 compute budget; num_persistent_tokens 是 schedule-time commit 预期; num_new_computed_tokens 是 update-time commit 事实——三者都是 int, 但语义截然不同。

commit 数字在不同时机的语义
DiT non-final step 不产生 reusable context / KV, 因此 num_persistent_tokens = 0、num_new_computed_tokens = 0。 DiT final step 默认 commit image context tokens / KV——原生支持 multiturn; 即使当前 response 后马上结束, 也不把"不 commit"作为默认语义。
13
§ 13 · ATTENTION 实现

Causal paged + DiT chunk bidirectional

第一版只保留一种方案: 复用 vLLM causal paged attention, 再补 DiT chunk 内 bidirectional attention, 最终在 softmax / logsumexp 语义下合并两段。

UAD attention semantics
不单独引入 attention_signature 字段。第一版由 phase、input_kind、 DiT chunk range、slot mapping 和 shape metadata 推导 attention 行为——保持 metadata 最小化。
14
§ 14 · EXECUTOR / RUNNER

拆 Attention 可以, 不要拆 FFN/MoE

UADInputs 由 runner 组装, 内容因 work 类型而异。executor 在合批策略上做出明确取舍: attention 可以拆, FFN/MoE 只要兼容就合批。

Work输入
AR token ids / positions / block tables / slot mapping
DiT latent or image-query embeds、timestep、positions、context KV / block tables、DiT chunk ranges
artifact VAE / artifact decode 所需 handle 或 latent
ATTENTION

可以按约束拆 kernel

AR causal、DiT chunk bidirectional 是两种 attention 模式, 合并到一个 kernel 可能不经济, 拆开是合理的。

FFN / MoE

能合批就合批

只要 hidden size、dtype、layer、TP / EP mesh 和 backend 兼容, AR hidden 和 DiT hidden 应在兼容层上形成更大的 FFN / MoE batch——这是 unified engine 的吞吐主体。

Output 处理规则:

  • AR sample 输出进入 v1 sampled-token 路径。
  • DiT denoise 输出进入 UADPhaseOutput.raw_output 和 runtime_state_delta。
  • final DiT context commit 通过 new_engine_tokens 和 num_new_computed_tokens 推进 UAD / v1 reusable context。
  • artifact decode 输出进入 new_materialized_tokens, 用于 streaming 或最终 response。

§ 15 · HUNYUANIMAGE3 FLOW

第一目标模型的 端到端

把所有概念串到 HunyuanImage3: text → image-emitting AR → DiT denoise → artifact decode → multiturn 继续。

HunyuanImage3 request 全生命周期
整条 pipeline 把 §05–§12 串了一遍: AR sample → image transition → DiT loop → final commit → artifact → 下一轮可见。 <img_ratio_*> 是 AR→DiT 的具体 trigger; final step 的 KV commit 是 multiturn 能跨轮看到 image context 的前提。

§ 16 · TODO

Scaffold → 真实 serving 还差什么

当前 11 个单元测试已经验证 scaffold 路径接通; 从 scaffold 到 production serving 需要逐项实现下面这些 boundary。

UAD state
连接 UADRequestState 和真实 v1 request lifecycle

当前 scaffold 没有把 UAD state 接到 v1 add_request / abort。这是让 UAD step 实际看到请求的前提。

phase trigger
实现 <img_ratio_*> 到 DiT phase 的 transition

AR sampler 输出 image-trigger token 时, 把 phase 切到 DIT_STEP 并填 runtime_state。

scheduling
AR / DiT scheduler budget、shape bucket、step batching

实现真正的合批策略, 让 token-like budget 在两类 work 之间被一致比较。

runner
DiT executor / worker / model runner

当前 UADExecutor 是 no-op scaffold; 需要实际的 DiT forward path。

attention
Causal paged + chunk bidirectional attention metadata / kernel

§13 描述的 reference 实现 + 高性能 kernel 替换。

KV commit
Final image context KV commit

DiT 最后一步要把 image context tokens / KV 持久化, 这样后续 AR 能复用。

output
Artifact output processor

VAE decode / artifact materialization 进入对外 ledger。

parallel
CFG parallel、SP / sequence parallel、AR + DiT FFN/MoE hidden 合批

从 unified engine 拿到吞吐红利的关键。需要更深的设计讨论。

UAD 不是一次性大重构, 而是分层落地: 先让 step path 接通, 再逐步让 phase、attention、commit、合批每一项都达到 production 质量。 — PROGRESS.md 五次推进路径

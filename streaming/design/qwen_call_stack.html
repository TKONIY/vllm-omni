vLLM-Omni · Architecture Notes

QwenOmni 与 QwenTTS 的
调用栈剖析

从 AsyncOmni.generate() 一路向下，到达 RVQ 码本、声码器、最终的语音波形。 本文档梳理两条多阶段流水线 —— 三阶段 Thinker → Talker → Code2Wav 全模态推理， 与轻量级两阶段 Talker → Code2Wav 文本到语音 —— 在 vLLM-Omni 中的代码路径与关键组件。

AR Talker Non-AR Code2Wav Multimodal Thinker Streaming Audio
Overview

统一的 多阶段 编排框架

vLLM-Omni 把任意多模态模型抽象成一条由若干 Stage 串接的流水线。 每个 Stage 拥有独立的 Engine / Executor / ModelRunner，由 Orchestrator 驱动跨阶段的张量传递。QwenOmni 与 QwenTTS 共享相同的编排骨架，只在阶段数量和模型类上有所不同。

01

入口编排

AsyncOmni 暴露 generate() 异步接口， 内部由 AsyncOmniEngine 启动后台 Orchestrator 线程，按 Stage 顺序把请求转交给对应的 StagePool。

02

阶段执行

每个 Stage 持有若干 StagePool 副本，副本内部是一个独立的 vLLM 引擎实例 —— 自回归阶段使用 GPUARModelRunner， 非自回归阶段使用 GPUGenerationModelRunner。

03

跨阶段载荷

阶段之间通过 StageSubmissionMessage 与 OmniPayload 传递隐藏状态、codes、参考音色等张量； OmniRequestState.mm_accumulated 在跨阶段过程中累积多模态输出。

Terminology

先把 名词 对齐

在追踪一条请求的旅程之前，先把 vLLM-Omni 里频繁出现、却容易和上游 vLLM 混淆的概念定义清楚。

Stage
流水线的一个逻辑节点。一个模型可被拆为多个 Stage —— QwenOmni 三个、QwenTTS 两个。 每个 Stage 拥有独立的 VllmConfig、KV cache、Scheduler 与 ModelRunner。
StagePool / StagePoolClient
同一 Stage 的副本池（一个副本 = 一个 EngineCore 子进程）。 StagePoolClient 抽象了 LLM 与 Diffusion 两种 stage 的统一访问接口。
Orchestrator
跨阶段调度器，运行在 AsyncOmni 主进程的后台线程里。 它持有所有 StagePool，按 PipelineConfig.stages 的拓扑 把上游输出转发给下游。
Thinker · Talker · Code2Wav
QwenOmni 的三个 Stage 角色： Thinker 做多模态理解 + 文本生成； Talker 把文本 hidden 转成 16 层 RVQ codes； Code2Wav 把 codes 解码成语音波形。QwenTTS 没有 Thinker。
RVQ codes · codebook size
Residual Vector Quantization。每一帧由 16 个 codebook 索引表示 （[T, 16]），codebook_size=2048。 超过该值的 token 视作控制符（如 EOS=2150）。
talker_mtp
Talker 内嵌的 Multi-Token-Predictor，单次前向预测 16 层 codes。 独立于 backbone 单独 CUDA Graph 捕获，是 Talker 阶段的性能关键路径。
StageExecutionType
LLM_AR：自回归生成，使用 GPUARModelRunner + OmniARScheduler。
LLM_GENERATION：一次性生成（如 Code2Wav），使用 GPUGenerationModelRunner + OmniGenerationScheduler。
OmniPayload · CodesStruct
跨阶段张量载荷的 TypedDict / msgspec.Struct 定义。 CodesStruct.audio 形状 [T, 16]，CodesStruct.ref 是可选的参考音色 codes。
mm_accumulated
OmniRequestState 的字段，dict[str, Any]。 ModelRunner 每步产出的 pooling_output 都会被按 modality 名累加到这里， 最终挂到 CompletionOutput.multimodal_output 上。
OmniTokensPrompt
Stage Input Processor 的输出类型 —— 把上游的 codes / hidden 重新打包为可投递下游 EngineCore 的 token prompt。 prompt_token_ids 是 codes flatten 后的整型序列。
StageSubmissionMessage
跨阶段控制消息（msgspec 序列化）。携带 request_id、prompt、 sampling_params_list、final_stage_id 等。 实际张量则走 EngineCoreRequest 的 additional_information。
PipelineConfig / StagePipelineConfig
声明式的流水线 dataclass。StagePipelineConfig.input_sources 描述上游依赖， 三类 processor 函数（process_input、process_next_stage_input、 async_chunk_process_next_stage_input）都以字符串路径登记。
final_output_type
每个 Stage 可选标记自己是 "text" 还是 "audio" 的最终输出。 FinalOutputModalityType 据此决定 AsyncOmni.generate() yield 哪些 chunk。
EngineCore
上游 vLLM 的术语，指真正持有模型与 KV cache 的工作进程。 在 vLLM-Omni 中每个 Stage 副本就是一个 StageEngineCoreProc（spawn 出来的子进程）。
OmniKVTransferManager
包装 vLLM KV cache 的管理器；在 async-chunk 路径下负责把上游已产 KV 块「过户」给下游。 相关 Scheduler 是 OmniARScheduler。
Suppressed-token mask
Talker 采样时对 logits 加的 mask，用以禁止生成不在 codec 字典内的非法 token —— 配合 stop_token_ids: [2150] 控制结束位置。
Receiving a Request

HTTP 到 EngineCore · 请求 是怎样落到模型上的

从用户的一次 POST /v1/chat/completions 出发，沿着 FastAPI 路由、AsyncOmni、 OmniInputPreprocessor、Orchestrator，一路追到第一个 Stage 的 EngineCore 子进程。

    HTTP 入口 · OpenAI 兼容路由

    vLLM-Omni 复用 vLLM 的 build_openai_app()，但把 /v1/chat/completions、/v1/audio/speech 等 multimodal 路由 从原始 router 上摘掉，再挂上自己的 handler。

    OmniOpenAIServingChat.create_chat_completion() 与 OmniOpenAIServingSpeech 都把请求转化为内部 prompt 结构后， 调用 engine_client.generate() —— 即 AsyncOmni.generate()。

    OmniInputPreprocessor._process_text() 走标准 tokenizer 路径； 如果 multi_modal_data 非空，再交给 _process_multimodal() 调用 HuggingFace 模型自带的 processor （图像 / 音频特征提取），结果合并成统一的 EngineCoreRequest。

    AsyncOmni.generate() · 包装异步生成器

    构造 OmniRequestState（含 mm_accumulated={} 初值）， 把 StageSubmissionMessage(type="add_request") 投递到 request_async_queue，然后开始 yield 来自 output_async_queue 的输出片段。

    Orchestrator 接管 · 派发到第一个 Stage

    后台 Orchestrator 线程从 request_async_queue 拉取消息， 根据 PipelineConfig.stages[0] 选定入口 Stage， 通过 StagePool.submit_initial() 把 EngineCoreRequest 投递给某个副本。

流式与 CUDA Graph 优化要点

分块解码
_decode_chunk_frames=300，_decode_left_context_frames=25， 通过左侧上下文衔接相邻 chunk，保持声学连续性。
桶式批处理
_get_decode_batch_bucket_frames() 将不同长度请求分到固定桶位， 降低 padding 浪费、增加可复用 graph 数量。
CUDA Graph
_capture_talker_mtp_graphs()（gpu_model_runner.py:196-251） 为 Talker 的 MTP code predictor 捕获多 batch_size 的 graph， 解码循环中按需切换。
异步增量
talker2code2wav_async_chunk 在 Talker 还在生成时即增量驱动 Code2Wav， 实现「边生成边出声」的低首包延迟。

def talker2code2wav(source_outputs, prompt=None, _requires_multimodal_data=False):
code2wav_inputs: list[OmniTokensPrompt] = []
for talker_output in source_outputs:
if not talker_output.finished: continue

return code2wav_inputs

QwenOmni / QwenTTS 在调度粒度上与上游 vLLM 一致；差异在于「decode 出来的 token 是 codes 而非可读字」， 以及 engine_output_type="latent" 时不做 detokenize。

④ Sampler · 停止条件 · 抑制 token

采样器
三个 Stage 都复用 vLLM 标准 sampler（top-p / top-k / temperature 等）。 采样发生在 ModelRunner 的 execute_model() 末尾， 模型类自身不持有 sampler 实例。
Thinker 停止
常规文本停止条件 —— EOS / max_tokens / stop strings，由用户的 SamplingParams 控制。
Talker 停止
stop_token_ids=[2150]，在 PipelineConfig 的 Talker 阶段 sampling_constraints 中声明。生成到 2150 即结束。
Talker · 抑制 token
talker_preprocess() 会构造 suppressed-token mask， 把不在 codec 字典内（≥ codebook_size=2048，除控制符外）的 logits 置为 -inf，保证采样落到合法 RVQ codes 上。
后置过滤
即便采到了 2150，Input Processor 还会做一次 frame-level 校验， qwen3_tts.talker2code2wav 中： valid_mask = audio_codes.any(dim=1) & (audio_codes.max(dim=1).values < 2048)， 丢弃任何越界帧，避免 Code2Wav 解码出噪声。
Code2Wav 采样
无 sampling。execution_type=LLM_GENERATION， ModelRunner 一次前向就把整段 codes 解成 waveform，detokenize=True 仅用来触发文本路径。

⑤ KV cache 的所有权

每个 Stage 副本运行在独立的 EngineCore 进程里，因此 各自持有一份 KV cache pool。 Thinker 的 KV cache 在 Thinker 进程内复用；Talker / Code2Wav 同理。 跨阶段的 KV「过户」只有在 async-chunk 模式下、由 OmniKVTransferManager 与 OmniARScheduler.chunk_transfer_adapter 联动完成。

① 内部输出对象 · OmniRequestOutput

End-to-End

把一切串起来 · 一条请求 的旅程

用 QwenTTS 上的一个典型 TTS 请求做完整 walkthrough： POST /v1/audio/speech，input="你好，世界"，voice="alice"。 每一格都标注了进程边界、关键调用、和在该步骤"流动的数据形状"。

  1. 1

    HTTP 入口

    FRONTEND FastAPI 路由命中 /v1/audio/speech，进入 OmniOpenAIServingSpeech.create_speech()。

    in: JSON · {input: "你好，世界", voice: "alice", ...}
  2. 2

    文本预处理 / 分词

    FRONTEND OmniInputPreprocessor 走 tokenizer，把 "你好，世界" → prompt_token_ids: list[int]， 打包成 EngineCoreRequest。

    out: EngineCoreRequest(prompt_token_ids=[...], sampling_params=..., additional_information={voice:"alice"})
  3. 3

    AsyncOmni → Orchestrator 队列

    FRONTEND AsyncOmni.generate() 创建 OmniRequestState，把 StageSubmissionMessage(type="add_request", final_stage_id=1) put 进 request_async_queue。

    channel: janus.AsyncQueue[EngineQueueMessage]
  4. 4

    Orchestrator 调度到 Stage 0 (Talker)

    FRONTEND Orchestrator 线程取出消息，按 QWEN3_TTS_PIPELINE.stages[0] 选定 Talker，调用 StagePool[0].submit_initial() → 内部 client.add_request_async(request) 经 ZMQ 推到副本子进程。

    channel: ZMQ DEALER → ROUTER · ipc:///tmp/...
  5. 5

    Talker EngineCore · Scheduler 入队

    TALKER · SUBPROC OmniARScheduler.add_request() 把请求放进 waiting 队列；下一个 step 由 schedule() 决定是否升级到 running。

    state: waiting deque · running list (vLLM 标准 V1 调度)
  6. 6

    Prefill · Talker 前向

    TALKER · SUBPROC GPUARModelRunner.execute_model() 一次喂入全部 prompt token， 产出 text_hidden_states；如果带参考音色，ECAPA_TDNN 把 voice="alice" 对应的参考 wav 编码成 speaker embedding。

    shape: hidden: [num_tokens, hidden_size], speaker_emb: [1, D]
  7. 7

    Decode · talker_mtp 出 16 层 codes

    TALKER · SUBPROC 每个 decode step：talker_mtp(hidden, speaker_emb) 单次输出 16 层 codes（CUDA Graph 加速）；vLLM Sampler 应用 suppressed-token mask，落到合法 codec id。 采到 2150 触发 stop。

    per-step: codes: [1, 16] · 累计写入 mm_accumulated["audio"]
  8. 8

    Talker 输出回流 · multimodal_output

    TALKER · SUBPROC 每个 step MultimodalOutputProcessor.process_outputs() 拦截 pooling_output，add_multimodal_tensor() 累加；最终 _new_completion_output() 把 mm_accumulated 挂上 CompletionOutput.multimodal_output。

    channel: ZMQ → 主进程的 StagePool 输出队列
  9. 9

    Stage 0 → 1 转发 · 两条路径取其一

    PATH A · SYNC Orchestrator 轮询到 Talker finished，调用 next_client.process_engine_inputs([output], prompt, streaming_ctx) → qwen3_tts.talker2code2wav() 过滤越界帧、reshape codes → 返回 list[OmniTokensPrompt] → ZMQ 投递给 Code2Wav。

    PATH B · ASYNC-CHUNK QwenTTS 默认走这条 —— Talker 每 step 末尾，scheduler 自己调 chunk_transfer_adapter.save_async()， 后台线程跑 talker2code2wav_async_chunk()， SharedMemoryConnector.put() 推到 Code2Wav。 Orchestrator 完全不参与本步。

    transform: [T, 16] → flat list[int] · 通道 A=ZMQ / B=共享内存

    Code2Wav · 一次前向出波形

    CODE2WAV · SUBPROC OmniGenerationScheduler 把整段 codes 一次性 prefill； Qwen3TTSCode2Wav.forward() 内部按 _decode_chunk_frames=300, _decode_left_context_frames=25 分块跑 Qwen3TTSTokenizerV2Decoder，输出 [B, samples] 波形。

    shape: waveform: [1, sample_count] · sr = _output_sample_rate

    CODE2WAV · SUBPROC 和 Talker 类似的累积路径，但这是 final_output_type="audio" 的终点 stage， FinalOutputModalityType 决定该 yield 给前端。

    channel: ZMQ → 主进程

Design Patterns

共享的 抽象 与差异点

两个模型共用一套阶段编排骨架，但在阶段数、输入模态与运行器组合上各有特化。

QwenOmni

  • 三阶段流水线，支持 text / audio / vision 多模态输入
  • Thinker 保留多层 hidden states 传给 Talker，实现 LM 状态共享
  • Talker 内置 MTP code predictor，一次预测 16 层 RVQ codes
  • Code2Wav 复用 Qwen3TTSTokenizerV2Decoder
  • 关注「理解 + 生成」的端到端语义一致性

QwenTTS

  • 两阶段轻量管道，输入仅为文本 + 可选参考音色
  • ECAPA-TDNN 提供 speaker-aware 合成能力
  • Code2Wav 采用 chunk + left-context 的流式解码
  • 桶式 batching 配合 CUDA Graph，主打高并发低延迟
  • 关注首包延迟与吞吐，常用于在线语音服务

通用设计模式

模式说明
统一模型壳 同一个 XxxForConditionalGeneration 类通过 model_stage 参数切换为 Thinker / Talker / Code2Wav 子模型，简化注册与权重加载。
Stage Input Processor 声明式输入转换函数（stage_input_processors/），把上游 Stage 的输出整形为下游 Stage 期望的 payload。

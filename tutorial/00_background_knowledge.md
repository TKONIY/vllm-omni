## 1. 大语言模型推理基础

### 1.1 自回归解码（Autoregressive Decoding）

大语言模型（LLM）以自回归的方式生成文本：每一步根据已有的所有 token 预测下一个 token，直到生成结束标记或达到最大长度。

```
输入: "今天天气"

步骤1: "今天天气" -> 预测 "真"
步骤2: "今天天气真"  -> 预测 "好"
步骤3: "今天天气真好" -> 预测 "<EOS>"

输出: "真好"
```

这一过程分为两个阶段：

- **预填充阶段（Prefill）**：一次性处理所有输入 token，计算它们的注意力表示。这是计算密集型操作。
- **解码阶段（Decode）**：逐个生成新 token，每步只计算新 token 与所有历史 token 的注意力。这是访存密集型操作。

### 1.2 KV Cache

在自回归解码中，每生成一个新 token 都需要与之前所有 token 做注意力计算。如果每次都重新计算，开销巨大。KV Cache 的核心思想是：**缓存已计算过的 Key 和 Value 矩阵**，避免重复计算。

```
                    注意力计算
                    ==========

  Query (当前token)
       |
       v
  Q * K^T / sqrt(d)  -->  softmax  -->  * V  -->  输出
       ^                                 ^
       |                                 |
   K Cache (所有历史token的Key)     V Cache (所有历史token的Value)
```

KV Cache 随序列长度线性增长，是 GPU 显存的主要消耗者之一。如何高效管理 KV Cache 是推理引擎的核心挑战。

### 1.3 注意力机制（Attention）

Transformer 的核心是多头注意力（Multi-Head Attention）。常见的变体包括：

| 类型 | 说明 | 典型模型 |
|------|------|----------|
| MHA（Multi-Head Attention） | 每个头都有独立的 K、V | GPT-2、LLaMA-1 |
| MQA（Multi-Query Attention） | 所有头共享一组 K、V | PaLM |
| GQA（Grouped-Query Attention） | 若干头分组共享 K、V | LLaMA-2/3、Qwen |

GQA 在推理效率和模型质量之间取得了良好平衡，是当前主流选择。

## 2. 多模态 AI

### 2.1 什么是多模态

多模态 AI 指能够处理和生成多种数据类型（模态）的模型。vLLM-Omni 支持以下四种模态：

```
    +---------+     +---------+     +---------+     +---------+
    |  文本   |     |  图像   |     |  视频   |     |  音频   |
    | (Text)  |     | (Image) |     | (Video) |     | (Audio) |
    +---------+     +---------+     +---------+     +---------+
         |               |               |               |
         v               v               v               v
    +-----------------------------------------------------------+
    |              多模态模型 (Omni Model)                       |
    +-----------------------------------------------------------+
         |               |               |               |
         v               v               v               v
    +---------+     +---------+     +---------+     +---------+
    |  文本   |     |  图像   |     |  视频   |     |  音频   |
    +---------+     +---------+     +---------+     +---------+
```

### 2.2 模态转换的基本范式

不同模态之间的转换通常需要编码器（Encoder）和解码器（Decoder）：

- **理解（Comprehension）**：将非文本模态编码为模型可理解的表示
  - 图像 -> Vision Encoder -> 视觉 token
  - 音频 -> Audio Encoder -> 音频 token
- **生成（Generation）**：将模型输出解码为目标模态
  - 隐向量 -> Diffusion Model -> 图像/视频
  - 语音 token -> Vocoder -> 音频波形

### 2.3 vLLM-Omni 支持的代表性模型

| 类别 | 模型 | 能力 |
|------|------|------|
| 全模态 | Qwen3-Omni | 文本/图像/音频输入 -> 文本/音频输出 |
| TTS | Qwen3-TTS, Fish Speech, CosyVoice3 | 文本 -> 语音 |
| 图像生成 | Flux, SD3, Hunyuan | 文本 -> 图像 |
| 视频生成 | Wan2.2, LTX2 | 文本 -> 视频 |
| 多模态理解 | Bagel | 图像/文本输入 -> 文本/图像输出 |

## 3. 扩散模型

### 3.1 基本原理

扩散模型（Diffusion Model）的核心思想是学习一个**逐步去噪**的过程：

```
前向过程（训练时）: 干净图像 -> 加噪 -> 加噪 -> ... -> 纯噪声
                     x_0       x_1      x_2           x_T

逆向过程（推理时）: 纯噪声 -> 去噪 -> 去噪 -> ... -> 干净图像
                     x_T      x_{T-1}  x_{T-2}        x_0
```

训练时，模型学习预测每一步的噪声；推理时，从随机噪声出发，逐步去噪还原出目标内容。

### 3.2 Diffusion Transformer (DiT)

传统扩散模型使用 U-Net 作为去噪网络。DiT（Diffusion Transformer）则用 Transformer 替代 U-Net，带来了更好的可扩展性和生成质量。当前主流图像/视频生成模型（Flux、SD3、Hunyuan 等）均采用 DiT 架构。

```
    +------------------+
    |   随机噪声 x_T   |
    +--------+---------+
             |
             v
    +--------+---------+      +------------------+
    |  DiT Transformer |<-----|  条件信息 (文本   |
    |  (去噪网络)      |      |  embedding 等)   |
    +--------+---------+      +------------------+
             |
             v
    +--------+---------+
    |   预测噪声 eps   |
    +--------+---------+
             |
     Scheduler 更新
             |
             v
    +--------+---------+
    |   x_{T-1}        |  <-- 重复 T 次
    +------------------+
             |
            ...
             |
             v
    +------------------+
    |   生成结果 x_0   |
    +------------------+
```

### 3.3 调度器（Scheduler）

调度器控制去噪过程的步数和每步的噪声强度。常见的调度器包括：

- **DDPM**：原始调度器，需要上千步去噪
- **DDIM**：确定性采样，可减少到 50-100 步
- **Flow Matching**：SD3、Flux 等新模型使用的调度方式，将去噪建模为从噪声到数据的流

步数越少速度越快，但质量可能下降。实际部署中通常使用 20-50 步。

### 3.4 扩散模型推理流水线

在 vLLM-Omni 中，扩散模型推理由 `DiffusionEngine` 管理：

```
    +-------------------------------------------------------------------+
    |                    Diffusion 推理流水线                             |
    +-------------------------------------------------------------------+
    |                                                                     |
    |   用户请求                                                          |
    |      |                                                              |
    |      v                                                              |
    |   文本编码 (Text Encoder)                                           |
    |      |                                                              |
    |      v                                                              |
    |   +-------------------------------------------+                     |
    |   |  去噪循环 (Denoising Loop)                |                     |
    |   |                                           |                     |
    |   |  for t in [T, T-1, ..., 1]:               |                     |
    |   |      noise_pred = DiT(x_t, t, condition)  |                     |
    |   |      x_{t-1} = scheduler.step(noise_pred) |                     |
    |   |                                           |                     |
    |   +-------------------------------------------+                     |
    |      |                                                              |
    |      v                                                              |
    |   VAE 解码 (Latent -> Pixel)                                       |
    |      |                                                              |
    |      v                                                              |
    |   输出图像/视频                                                     |
    |                                                                     |
    +-------------------------------------------------------------------+
```

## 4. 语音合成（TTS）

### 4.1 基于编解码器的 TTS

现代 TTS 系统（如 Qwen3-TTS、Fish Speech、CosyVoice3）普遍采用**编解码器（Codec）**方案，将连续音频信号离散化为语音 token，从而可以复用 LLM 的自回归框架。

```
    训练时:
    语音波形 --> Codec Encoder --> 离散语音 token (codebook indices)

    推理时:
    文本 --> LLM 自回归生成语音 token --> Codec Decoder (Vocoder) --> 语音波形
```

### 4.2 语音 Token 与 Codebook

音频编解码器（如 SpeechTokenizer、DAC）使用残差向量量化（RVQ），将音频压缩为多层离散 token：

```
    连续音频帧
         |
         v
    +----+----+----+----+----+----+
    | 编码器（Encoder）           |
    +----+----+----+----+----+----+
         |
         v
    第1层量化: [45, 12, 78, 23, ...]   <-- 粗粒度（韵律、基频）
    第2层量化: [3,  90, 15, 67, ...]   <-- 中粒度（音色细节）
    第3层量化: [56, 34, 82, 11, ...]   <-- 细粒度（高频细节）
    ...
```

LLM 通常只预测第 1 层 token（称为 semantic token），后续层由专门的模型补全。

### 4.3 Vocoder

Vocoder 是将语音 token（或频谱特征）转换回音频波形的解码器。在 vLLM-Omni 中，这一步通常由 `code2wav` 阶段完成。

### 4.4 vLLM-Omni 中的 TTS 流水线

以 Qwen3-TTS 为例，其推理流程被拆分为两个阶段：

```
    用户输入 (文本 + 参考音频)
              |
              v
    +-----------------------------+
    | Stage 0: Talker (LLM)       |
    | - 自回归生成语音 token       |
    | - 输出: 离散语音编码         |
    +-----------------------------+
              |
        (SharedMemory Connector)
        (流式传输，逐 chunk)
              |
              v
    +-----------------------------+
    | Stage 1: Code2Wav           |
    | - 将语音 token 解码为波形    |
    | - 支持流式 chunk 解码        |
    | - 输出: 音频数据             |
    +-----------------------------+
              |
              v
         音频输出
```

## 5. 流水线/阶段架构

### 5.1 为什么需要阶段化

全模态模型通常由多个异构子模型组成（如 Thinker + Talker + Vocoder），它们具有不同的计算特征：

- **自回归模型（AR）**：逐 token 生成，需要 KV Cache 管理和连续批处理
- **扩散模型（Diffusion）**：迭代去噪，需要 GPU 计算密集操作
- **非自回归生成模型（Generation）**：一次性前向推理（如 Vocoder）

将它们放在同一个推理引擎中管理会非常低效。vLLM-Omni 的解决方案是：**将模型拆分为多个阶段（Stage），每个阶段独立调度和执行**。

### 5.2 pipeline.yaml 配置

每个模型的阶段划分通过 `pipeline.yaml` 配置文件定义。以下是配置的核心结构：

```yaml
stages:
  - stage_id: 0           # 阶段编号
    stage_type: llm        # 阶段类型: llm 或 diffusion
    worker_type: ar        # Worker 类型: ar（自回归）或 generation
    input_sources: []      # 输入来源（空表示来自用户请求）
    final_output: false    # 是否为最终输出阶段

  - stage_id: 1
    stage_type: llm
    worker_type: generation
    input_sources: [0]     # 接收 stage 0 的输出
    final_output: true     # 最终输出阶段

connectors:               # 阶段间连接器配置
  connector_of_shared_memory:
    name: SharedMemoryConnector

edges:                    # 阶段间的数据流边
  - from: 0
    to: 1
```

### 5.3 多阶段推理流水线

以 Qwen3-Omni（文本+语音输出）为例，完整的多阶段推理流水线如下：

```
    用户请求 (文本/图像/音频)
              |
              v
    +-------------------------------+
    |  Stage 0: Thinker (大语言模型) |
    |  - 理解多模态输入              |
    |  - 自回归生成文本 token        |
    |  - 类型: LLM / AR             |
    +-------------------------------+
              |
         OmniConnector
              |
              v
    +-------------------------------+
    |  Stage 1: Talker (语音预测)    |
    |  - 接收 Thinker 的隐状态       |
    |  - 生成语音 token (codec)      |
    |  - 类型: LLM / AR             |
    +-------------------------------+
              |
         OmniConnector
              |
              v
    +-------------------------------+
    |  Stage 2: Code2Wav (声码器)    |
    |  - 将语音 token 解码为波形     |
    |  - 类型: LLM / Generation     |
    +-------------------------------+
              |
              v
    最终输出 (文本 + 音频)
```

### 5.4 Orchestrator 与 StageEngineCoreClient

vLLM-Omni 使用 **Orchestrator（编排器）**来协调多个阶段的执行：

```
    +---------------------------------------------------------------+
    |                     AsyncOmniEngine                            |
    |                                                                |
    |   +-------------------------------------------------------+   |
    |   |                  Orchestrator                          |   |
    |   |                                                       |   |
    |   |   +------------------+    +------------------+        |   |
    |   |   | StageEngineCore  |    | StageEngineCore  |        |   |
    |   |   | Client (Stage 0) |    | Client (Stage 1) |  ...   |   |
    |   |   +--------+---------+    +--------+---------+        |   |
    |   |            |                       |                  |   |
    |   +-------------------------------------------------------+   |
    |                |                       |                      |
    |                v                       v                      |
    |        +-------+--------+      +-------+--------+            |
    |        | Engine Core 0  |      | Engine Core 1  |            |
    |        | (独立进程)      |      | (独立进程)     |            |
    |        +----------------+      +----------------+            |
    +---------------------------------------------------------------+
```

- **Orchestrator** 运行在后台线程中，管理请求在阶段间的流转
- **StageEngineCoreClient** 负责与每个阶段的引擎核心通信
- **OmniConnector** 负责阶段间的数据传输（共享内存、网络等）

## 6. 分布式推理

### 6.1 并行策略

当模型过大无法放入单张 GPU，或需要提高吞吐量时，需要分布式推理。vLLM-Omni 支持以下并行策略：

```
    +-------------------------------------------------------+
    |               分布式并行策略                            |
    +-------------------------------------------------------+
    |                                                       |
    |  张量并行 (TP)      流水线并行 (PP)     数据并行 (DP)  |
    |  将单层切分到       将不同层分配到      多副本处理      |
    |  多个 GPU          不同 GPU            不同请求        |
    |                                                       |
    |  GPU0: 权重的前半   GPU0: Layer 0-15   GPU0: 模型副本A |
    |  GPU1: 权重的后半   GPU1: Layer 16-31  GPU1: 模型副本B |
    |                                                       |
    +-------------------------------------------------------+
```

#### 张量并行（Tensor Parallelism, TP）

将模型的每一层按列或行切分到多张 GPU 上，每张 GPU 计算部分结果后再汇总。适合单机多卡场景。

```
    输入 X
    |
    +------+------+
    |             |
    v             v
  GPU 0         GPU 1
  W[:, :n/2]    W[:, n/2:]     <-- 权重按列切分
    |             |
    v             v
  Y_0           Y_1
    |             |
    +------+------+
           |
     AllReduce 汇总
           |
           v
        输出 Y
```

#### 流水线并行（Pipeline Parallelism, PP）

将模型的不同 Transformer 层分配到不同 GPU 上，前一个 GPU 的输出作为后一个 GPU 的输入。适合跨机场景。

#### 数据并行（Data Parallelism, DP）

部署多个模型副本，不同请求由不同副本处理。在 vLLM-Omni 中，这通过 LoadBalancer 实现。

### 6.2 vLLM-Omni 的分布式执行模型

vLLM-Omni 在分布式方面做了扩展，以支持异构多阶段模型的分布式部署：

```
    +---------------------------------------------------------------+
    |                   分布式执行模型                                |
    +---------------------------------------------------------------+
    |                                                               |
    |   Node 0 (机器 0)                                             |
    |   +-------------------------+  +-------------------------+    |
    |   | GPU 0                   |  | GPU 1                   |    |
    |   | Stage 0 (Thinker, TP=2) |  | Stage 0 (Thinker, TP=2) |   |
    |   +-------------------------+  +-------------------------+    |
    |                                                               |
    |   Node 1 (机器 1)                                             |
    |   +-------------------------+  +-------------------------+    |
    |   | GPU 0                   |  | GPU 1                   |    |
    |   | Stage 1 (Talker)        |  | Stage 2 (Code2Wav)      |    |
    |   +-------------------------+  +-------------------------+    |
    |                                                               |
    |   OmniCoordinator: 跨节点协调阶段间的请求调度与数据传输         |
    |   OmniConnector:   阶段间的数据传输通道                        |
    |   LoadBalancer:    在同一阶段的多副本间做负载均衡               |
    |                                                               |
    +---------------------------------------------------------------+
```

关键组件：

- **OmniCoordinator**：全局协调器，负责跨阶段的请求路由和状态同步
- **OmniConnector**：阶段间的数据传输抽象，支持共享内存（单机）和网络传输（跨机）
- **KV Transfer Manager**：当 Thinker 和 Talker 分布在不同 GPU 时，负责传输 KV Cache

### 6.3 设备分配

每个阶段通过 `pipeline.yaml` 中的 `runtime.devices` 指定使用哪些 GPU。例如：

```yaml
stages:
  - stage_id: 0
    runtime:
      devices: "0,1"      # Thinker 使用 GPU 0 和 GPU 1（TP=2）
  - stage_id: 1
    runtime:
      devices: "2"         # Talker 使用 GPU 2
  - stage_id: 2
    runtime:
      devices: "3"         # Code2Wav 使用 GPU 3
```

## 7. vLLM 核心概念

vLLM-Omni 构建在 vLLM 之上，继承了 vLLM 的关键技术。以下是需要了解的核心概念。

### 7.1 PagedAttention

PagedAttention 是 vLLM 的核心创新。借鉴操作系统的虚拟内存分页机制，将 KV Cache 划分为固定大小的**页（Block）**，按需分配，避免内存浪费。

```
    传统方式:
    +-----------------------------------------------+
    | 请求A的KV Cache (预分配最大长度，大量浪费)       |
    +-----------------------------------------------+

    PagedAttention:
    +------+------+------+------+------+------+------+
    | Blk0 | Blk1 | Blk2 | Blk3 | Blk4 | Blk5 | Blk6 |
    +------+------+------+------+------+------+------+
      请求A   请求A   请求B   请求A   请求B   空闲    空闲

    - 物理块按需分配，无需连续
    - 不同请求的块可以交错存放
    - 空闲块可以立即被新请求使用
```

好处：
- 显存利用率接近 100%，几乎无碎片
- 支持高效的前缀缓存（Prefix Caching）：相同前缀的请求可共享 KV Cache 块

### 7.2 连续批处理（Continuous Batching）

传统批处理需要等一批请求全部完成后才能处理下一批。连续批处理则在每个解码步骤后检查是否有请求完成或新请求到达，动态调整批次组成。

```
    传统批处理:                     连续批处理:
    时间步 ->                      时间步 ->

    | A A A A A |                  | A B C C |
    | B B B B B | (B等A完成)       | A B C D |
    | C C C C C |                  | A E C D | (B完成，E加入)
                                   | F E C D | (A完成，F加入)

    请求B、C必须等请求A             新请求立即加入，
    全部完成后才能开始               已完成请求立即释放资源
```

### 7.3 引擎架构

vLLM 的 V1 引擎架构（vLLM-Omni 所基于的版本）主要组件如下：

```
    +----------------------------------------------------------+
    |                     vLLM V1 Engine                        |
    +----------------------------------------------------------+
    |                                                          |
    |  +-------------------+                                   |
    |  | InputProcessor    |  <-- 处理用户输入（分词、多模态）   |
    |  +-------------------+                                   |
    |           |                                              |
    |           v                                              |
    |  +-------------------+                                   |
    |  | Scheduler         |  <-- 决定当前步骤执行哪些请求       |
    |  +-------------------+                                   |
    |           |                                              |
    |           v                                              |
    |  +-------------------+                                   |
    |  | ModelRunner        |  <-- 执行模型前向计算             |
    |  +-------------------+                                   |
    |           |                                              |
    |           v                                              |
    |  +-------------------+                                   |
    |  | OutputProcessor   |  <-- 后处理（解码token、汇总结果）  |
    |  +-------------------+                                   |
    |                                                          |
    +----------------------------------------------------------+
```

### 8.2 两类引擎

vLLM-Omni 内部管理两类推理引擎：

| 引擎类型 | 用途 | 模型类型 | 调度方式 |
|----------|------|----------|----------|
| **EngineCore**（基于 vLLM） | 自回归生成 | LLM、TTS Talker、Vocoder | 连续批处理 |
| **DiffusionEngine** | 扩散模型推理 | DiT（图像/视频生成） | 请求级调度 |

### 8.3 Worker 类型

每个 LLM 阶段的实际计算由 Worker 执行。vLLM-Omni 定义了以下 Worker 类型：

- **AR Worker**（`gpu_ar_worker.py`）：用于自回归模型，支持 KV Cache 管理和连续批处理
- **Generation Worker**（`gpu_generation_worker.py`）：用于非自回归生成模型（如 Vocoder），一次前向计算完成生成

### 8.4 阶段类型总结

```
    +------------------+------------------+------------------+
    |    AR (自回归)    | Generation (生成) | Diffusion (扩散) |
    +------------------+------------------+------------------+
    | 逐token生成       | 一次前向完成      | 迭代去噪          |
    | 需要KV Cache      | 无需KV Cache     | 无需KV Cache      |
    | 连续批处理        | 简单批处理        | 请求级调度         |
    +------------------+------------------+------------------+
    | Thinker (思考)    | Code2Wav (声码)  | Flux (图像生成)   |
    | Talker (语音预测) |                  | SD3 (图像生成)    |
    | Fish Speech AR   |                  | Wan2.2 (视频生成) |
    +------------------+------------------+------------------+
```

## 小结

在深入 vLLM-Omni 的代码之前，理解以下核心概念将帮助你更好地阅读后续教程：

1. **自回归解码**与 **KV Cache** 是 LLM 推理的基础
2. **多模态**处理需要不同的编码器和解码器来桥接各模态
3. **扩散模型**通过迭代去噪生成高质量图像/视频/音频
4. **Codec-based TTS** 将语音离散化为 token，复用 LLM 框架生成语音
5. **阶段化架构**（pipeline.yaml）将异构子模型拆分为独立调度的阶段
6. **分布式推理**通过张量并行、流水线并行和数据并行实现大模型高效部署
7. **vLLM 的 PagedAttention 和连续批处理**是高效推理的核心技术
8. **Orchestrator** 负责协调多阶段的请求流转和数据传输

掌握这些背景知识后，你将能够理解 vLLM-Omni 的设计动机和实现细节。接下来的教程将逐步深入各个模块的源代码。
